# Ephemera: Mastodon Picture-Posting Bot Design

## Overview

A bot that posts pictures to Mastodon on a regular schedule. It runs in a single Docker container and stores its media in an S3-compatible file storage service.

## Architecture

Five components in a single Docker container (Debian stable):

- **ephemera** — Rails 8 application for managing the picture collection (localhost:3000)
- **postpic** — CLI tool (`bin/postpic`) for posting the next scheduled picture to Mastodon
- **register** — CLI tool (`bin/register`) for creating a Mastodon client app and completing OAuth
- **Mapi** — Shared Mastodon API client library in `lib/mapi/`
- **nginx** — Reverse proxy for SSL termination (:443 → :3000)

Installed in the container: Ruby 3.3.x, SQLite3, nginx, certbot, cron, exiftool.

### Configuration

- `config/mastodon.json` — Mastodon app name, server URL, client ID, client secret, access token, redirect URI. Written by `bin/register`. Gitignored (template at `config/mastodon.json.example`).
- Rails encrypted credentials — S3 access key, secret key, bucket, endpoint, region.
- Environment variable `EPHEMERA_DOMAIN` — domain name for nginx/certbot.

### Frontend

Rails 8 defaults: Hotwire (Turbo + Stimulus) with importmaps. No JavaScript build step.

### Testing

Minitest + Rails system tests. Specs written first for every component.

## Models and Database

### Picture

| Column | Type | Notes |
|---|---|---|
| id | integer | PK |
| s3_key | string | S3 object key |
| alt_text | text | Media attachment description for accessibility |
| hashtags | text | Space-separated hashtags, used as status text |
| sensitive_content | string | If non-blank: sets sensitive=true and spoiler_text on the Mastodon status |
| order | integer | Unique, gapless 1..N sequence |
| original_filename | string | Preserved from upload |
| content_type | string | MIME type |
| created_at / updated_at | datetime | Rails timestamps |

### Log

| Column | Type | Notes |
|---|---|---|
| id | integer | PK |
| picture_id | integer | FK to pictures |
| posted_at | datetime | When this picture was posted to Mastodon |
| posted_order | integer | The order value at time of posting |

### Auth models

Generated by Rails 8 `bin/rails generate authentication`: User, Session, Current.

### Ordering invariants

- `pictures.order` has a unique index.
- On create: `order = MAX(order) + 1` (or 1 if empty).
- On reorder (numeric input): shift affected pictures up or down to maintain gapless sequence.
- On delete: compact all subsequent orders (decrement) to close the gap.
- Invariant: `MAX(order) == COUNT(*)` at all times.

## Mapi Library

Located in `lib/mapi/`, contained in a Ruby module `Mapi`. Uses `Net::HTTP` (stdlib).

### Modules

- **Mapi::Client** — Core HTTP client. Loads credentials from `config/mastodon.json`. Provides `get`, `post`, `patch` methods with JSON handling and auth headers.
- **Mapi::Apps** — `POST /api/v1/apps` to register a new client application.
- **Mapi::OAuth** — Generates authorization URL (`urn:ietf:wg:oauth:2.0:oob` redirect for terminal flow), exchanges authorization code for access token.
- **Mapi::Accounts** — `GET /api/v1/accounts/verify_credentials`, `PATCH /api/v1/accounts/update_credentials`, `GET /api/v1/accounts/:id/followers`.
- **Mapi::Media** — `POST /api/v2/media` to upload media attachments with description.
- **Mapi::Statuses** — `POST /api/v1/statuses` to create a status with media IDs, text, sensitive flag, spoiler_text.

### OAuth scopes

`read:accounts read:follows write:media write:statuses write:accounts`

### Config file structure (config/mastodon.json)

```json
{
  "server": "https://mastodon.example.com",
  "app_name": "Ephemera",
  "client_id": null,
  "client_secret": null,
  "access_token": null,
  "redirect_uri": "urn:ietf:wg:oauth:2.0:oob"
}
```

## Command-Line Tools

### bin/register

Handles the full OAuth setup flow:

1. Reads `server` and `app_name` from `config/mastodon.json`
2. Calls `Mapi::Apps.create` → saves `client_id` and `client_secret`
3. Generates OAuth authorization URL, prints it
4. Prompts user to visit URL and paste back the authorization code
5. Exchanges code for access token via `Mapi::OAuth.token`
6. Saves `access_token` to `mastodon.json`
7. Verifies by calling `Mapi::Accounts.verify_credentials`

### bin/postpic

Posts the next scheduled picture:

1. Boots Rails environment
2. Finds last posted picture: `Log.order(posted_at: :desc).first`
3. Determines next order: `log.posted_order + 1` (or 1 if no log)
4. Finds picture at that order (wraps to order 1 if not found)
5. Downloads image from S3 to temp file
6. Uploads to Mastodon via `Mapi::Media.create` (with alt_text as description)
7. Creates status via `Mapi::Statuses.create` (hashtags as text, sensitive if applicable)
8. Writes Log entry
9. Deletes temp file

### Cron schedule

- **postpic:** Every 3 days at 10:11 AM Pacific — `11 10 */3 * *`
- **certbot:** 1st of every 2nd month at 3 AM — `0 3 1 */2 *` (well within 90-day cert lifetime)

## EXIF Processing

On upload, before storing in S3, images are processed with exiftool to strip potentially identifying metadata.

**Approach:** `exiftool -all= -tagsfromfile @ <allowed_tags> input.jpg`

**Allowlisted tags** (strictly image-technical):
- FNumber, ExposureTime, ISOSpeedRatings, FocalLength, FocalLengthIn35mmFormat
- ImageWidth, ImageHeight
- WhiteBalance, MeteringMode, ExposureProgram, ExposureCompensation
- Flash, LightSource, SceneCaptureType
- ColorSpace, BitsPerSample, Compression, PhotometricInterpretation
- Orientation, XResolution, YResolution, ResolutionUnit

**Explicitly forbidden:**
- All GPS tags
- All date/time tags
- Camera identification (Make, Model, SerialNumber, LensModel, etc.)
- Software tags
- Author/copyright tags
- Thumbnail data

## Controllers and Views

### PicturesController

Full CRUD. Index shows pictures in order with thumbnails. Create handles multi-file upload with EXIF stripping. Edit allows metadata changes and numeric order input for reordering. Destroy removes from S3 and compacts order.

### MastodonProfileController

Show fetches profile via Mapi. Edit/update patches profile via Mapi. Display name, bio, avatar, header.

### FollowersController

Index fetches and displays paginated followers list via Mapi.

### LogsController

Index shows posting history table: thumbnail, posted date/time, picture info. Read-only.

All controllers require authentication. Routes are standard RESTful resources behind an authenticated block. App layout includes navigation bar (Pictures, Profile, Followers, Logs, Logout).

## Docker and nginx

### Dockerfile

- Base: `debian:bookworm-slim`
- Installs: ruby, sqlite3, libsqlite3-dev, nginx, certbot, python3-certbot-nginx, cron, exiftool, build-essential, libffi-dev
- Copies app, runs `bundle install`, precompiles assets
- Exposes ports 80 and 443

### nginx

- Listens on 443 (SSL, managed by certbot) and 80 (redirect to 443 + ACME challenges)
- Proxies to `localhost:3000`
- Domain from `EPHEMERA_DOMAIN` env var (templated via envsubst at startup)

### Startup (bin/start.sh)

1. Start cron daemon
2. Start nginx
3. Run `db:migrate`
4. Start Puma on localhost:3000 (foreground — keeps container alive)

Container timezone: `America/Los_Angeles`
